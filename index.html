Section 01 — The bottleneck

Why HBM is the real constraint

The H100 has 80 GB of HBM (high-bandwidth memory) where model weights live, and only 50 MB of on-chip SRAM. Every computation requires weights to travel from HBM to SRAM — and that bandwidth pipe (3.35 TB/s) is what limits everything.

The memory bandwidth gap: Compute hardware has improved at roughly Moore's Law pace. Memory bandwidth has improved much more slowly. This divergence is what makes LLM inference structurally difficult — not a software bug, not a framework inefficiency.
Section 02 — Autoregressive generation

One token. One full weight transfer. Repeat.

An 8B model in 16-bit precision occupies 16 GB. Generating each new token requires a complete forward pass — all 16 GB of weights must travel from HBM to on-chip SRAM, be used briefly, then discarded for the next layer.

The 200× gap is structural: For a 1,000-token response you need 1,000 complete weight transfers. With 3.35 TB/s bandwidth and 16 GB weights per transfer: 3,350 ÷ 16 ≈ 200 transfers/second. The compute units capable of 1,000 TFLOP/s sit idle most of this time, waiting for data.
Section 03 — Batching & KV cache

Batching helps — until the KV cache kills it

Processing N queries simultaneously amortizes one weight transfer across N responses, pushing arithmetic intensity up. But KV cache memory scales with batch size × sequence length × model depth, consuming HBM and capping how large batches can grow.

PagedAttention (vLLM) borrows ideas from OS virtual memory to manage KV cache blocks non-contiguously, reducing fragmentation and enabling larger effective batch sizes. TensorRT-LLM adds fused kernels and multi-head attention optimizations. But they all operate within the same fundamental constraint.
Section 04 — Speculative decoding

Guess cheaply. Verify in parallel.

A small draft model proposes several token candidates cheaply. The large verifier model processes the entire drafted sequence in a single forward pass — similar to prefill where multiple tokens are handled simultaneously — and accepts or rejects each using a criterion that preserves the target distribution.

The calibration challenge: Too large a draft model → loses speed advantage. Too small → acceptance rates collapse and you do more total work than naive decoding. Acceptance rate is also sensitive to temperature, prompt distribution, and draft length. Variants: self-speculative decoding uses early exit layers of the same model; lookahead decoding uses n-gram speculation from the input.
Section 05 — Diffusion LLMs & Block Diffusion

Escape the autoregressive paradigm

Diffusion LLMs start with a fully masked sequence and iteratively refine all positions simultaneously. Each denoising step updates every token — arithmetic intensity scales with context length L (opposite of AR decode). Block Diffusion hybridizes: diffusion within fixed blocks, autoregressive across blocks.

Block diffusion (ICLR 2025 Oral) is the most promising direction: diffusion within blocks gives high arithmetic intensity; autoregressive across blocks enables early stopping at <EOS> and KV caching. Most AR optimizations (speculative decoding, paged attention, continuous batching) can be grafted onto block diffusion without major rearchitecting.
Section 06 — The roofline model

Where does each workload fall?

The roofline model (Williams et al., 2009) plots achieved performance against arithmetic intensity (FLOPs/byte). Below the ridge point: memory-bound — performance scales with bandwidth. Above: compute-bound — limited by peak FLOP/s. AR decode is always far left. Prefill and diffusion cross into compute-bound territory.

The economic argument: Architectures that shift workloads toward the compute-bound regime directly reduce cost per token by making better use of silicon that's already paid for. The 200× gap means you're paying for peak FLOP/s hardware but productively using only a small fraction. Solving this translates directly into cheaper, faster AI.
